# Ontology 学习知识空白清单

## 在撰写费曼解释时发现的知识空白

### 1. Ontology Engine 的技术实现细节 ⚠️

**当前理解**：
- 知道它有"读架构"和"写架构"
- 知道它可以处理百万级读写
- 知道它能跨异构系统工作

**知识空白**：
- ❓ Ontology Engine 底层用的是什么技术？
- ❓ 它如何实现"一个统一的现实"（单一事实来源）？
- ❓ 它如何保证跨多个数据源的事务一致性？
- ❓ 它的查询语言是什么？SQL？GraphQL？还是自定义的？

**为什么重要**：
- 理解技术实现有助于评估性能和局限性
- 对于技术决策者很重要

**补充研究方向**：
- 查看 Palantir 技术文档中关于 Ontology Engine 的架构说明
- 了解 OSDK 的 API 设计

---

### 2. Ontology 与传统知识图谱的区别 ⚠️

**当前理解**：
- Ontology 不仅是知识图谱，还包含动作和逻辑
- Ontology 是"可执行的"

**知识空白**：
- ❓ Ontology 和 Neo4j、Amazon Neptune 等图数据库有什么本质区别？
- ❓ Ontology 和 RDF/OWL（语义网标准）有什么关系？
- ❓ Palantir 是否使用了现有的本体论标准，还是完全自定义的？

**为什么重要**：
- 帮助理解 Palantir 的独特价值
- 避免将 Ontology 简单等同于知识图谱

**补充研究方向**：
- 对比 Palantir Ontology 和传统知识图谱的功能
- 了解 Ontology Language 的语法和设计哲学

---

### 3. Ontology 建模的最佳实践 ⚠️

**当前理解**：
- 知道要定义"名词"（Objects）和"动词"（Actions）
- 知道要建模数据、逻辑、动作、安全

**知识空白**：
- ❓ 如何决定哪些应该建模为 Object，哪些应该建模为 Property？
- ❓ 如何设计 Object 之间的关系（Links）？
- ❓ 如何避免过度建模或建模不足？
- ❓ 有没有行业最佳实践或设计模式？

**为什么重要**：
- 实际项目中需要做建模决策
- 好的建模是 Ontology 成功的关键

**补充研究方向**：
- 查找 Palantir 的 Ontology 建模教程
- 研究现有的 Ontology 案例（如 ShipOS 的本体模型）
